先把两个核心问题拆开说：

1. **SDP 本身是什么结构**
2. **URI 是 `xxx.mp4` 这种真实场景下，服务器是怎么“用文件元数据 → 生成 SDP”的**

------

## 一、SDP 大致长什么样？

SDP = 纯文本，按行组织，每行都是：

```text
<单个字母>=<内容>
```

大致可以分两层：

- **会话级（全局）**：描述整个会话的信息
- **媒体级（per track）**：每个 `m=` 开头的块描述一个 track（video/audio 等）

先看一个典型的 RTSP + H264 + AAC SDP 示例：

```text
v=0
o=- 0 0 IN IP4 127.0.0.1
s=Example MP4 Stream
c=IN IP4 0.0.0.0
t=0 0
a=control:*

m=video 0 RTP/AVP 96
a=rtpmap:96 H264/90000
a=fmtp:96 packetization-mode=1;profile-level-id=42C01E;
 sprop-parameter-sets=Z0LAHtkAoD2Q,aM48gA==
a=control:trackID=1

m=audio 0 RTP/AVP 97
a=rtpmap:97 MPEG4-GENERIC/44100/2
a=fmtp:97 streamtype=5; profile-level-id=15; mode=AAC-hbr;
 config=1190; SizeLength=13; IndexLength=3; IndexDeltaLength=3;
a=control:trackID=2
```

### 1. 会话级常见字段

这些在 SDP 开头、在第一个 `m=` 之前：

- `v=`：版本，固定 `v=0`

- `o=`：会话拥有者信息（谁发起的），一般服务器随便填：

  ```text
  o=- 0 0 IN IP4 127.0.0.1
  ```

- `s=`：会话名（可以认为是“标题”）

  ```text
  s=Example MP4 Stream
  ```

- `c=`：连接信息，IP 类型和地址

  ```text
  c=IN IP4 0.0.0.0       ; 0.0.0.0 表示“由 RTSP 来决定具体 IP”
  ```

- `t=`：时间

  ```text
  t=0 0                  ; 表示“永久有效”
  ```

- `a=control:*`：RTSP 相关
   表示整个会话的 control URL 是“base URL 本身”，下面每个 track 再有自己的 `control:trackID=x`。

### 2. 媒体级（每个 track）常见字段

每个 track 从一行 `m=` 开头：

#### `m=` 行

```text
m=<media> <port> <proto> <payload type(s)>
```

- `<media>`：`audio` / `video` / `application` 等
- `<port>`：对 RTSP 来说可以写 `0`（由 RTSP SETUP 的 Transport 决定端口）
- `<proto>`：`RTP/AVP`（基于 RTP 的 Audio-Video Profile）
- `<pt>`：**RTP payload type**，96+ 为动态 payload

上面例子：

```text
m=video 0 RTP/AVP 96
m=audio 0 RTP/AVP 97
```

代表：

- 有一个 video track，用 PT=96
- 有一个 audio track，用 PT=97

#### `a=rtpmap`

```text
a=rtpmap:<pt> <codec-name>/<clock rate>[/channels]
```

例子：

```text
a=rtpmap:96 H264/90000              ; 视频 H.264，RTP 时钟 90000
a=rtpmap:97 MPEG4-GENERIC/44100/2   ; 音频 AAC，44.1kHz，2 声道
```

clock rate：视频通常 90000，音频用其采样率（比如 44100）。

#### `a=fmtp`

**codec 专有参数**，解释 `payload type` 对应的一些附加信息。

- H.264 典型：

  ```text
  a=fmtp:96 packetization-mode=1;profile-level-id=42C01E;
   sprop-parameter-sets=Z0LAHtkAoD2Q,aM48gA==
  ```

  这里的 `sprop-parameter-sets` 就是 **SPS/PPS** 的 Base64 编码；
   `profile-level-id` 对应 H.264 Profile/Level。

- AAC 典型：

  ```text
  a=fmtp:97 streamtype=5; profile-level-id=15; mode=AAC-hbr;
   config=1190; SizeLength=13; IndexLength=3; IndexDeltaLength=3;
  ```

  `config=1190` 是 AudioSpecificConfig 的十六进制编码（从容器里的 extradata 推出来）。

#### `a=control`

RTSP 会特别用到：

```text
a=control:trackID=1
a=control:trackID=2
```

DESCRIBE 返回 SDP 后，客户端会根据这些 `control` 字段去发 SETUP：

- SETUP `rtsp://ip:port/xxx.mp4/trackID=1` → video
- SETUP `rtsp://ip:port/xxx.mp4/trackID=2` → audio

## 二、URI 是 xxx.mp4 时，真实场景怎么从“文件元数据”生成 SDP？

你的猜测 **“通过获取 mp4 的元数据来构建 SDP”**，在 VoD（点播）场景里基本是对的，真实服务器也是这么干，只是会再抽象一层，流程大概是：

> URI → 寻找媒体源 → 打开容器（MP4）→ 解析每个 track 的 codec/参数 → 映射成 SDP

### 1. URI 到“媒体源”的映射

举例：客户端发

```text
DESCRIBE rtsp://example.com:8554/movie.mp4 RTSP/1.0
```

服务器内部一般会做一层路由：

- `/movie.mp4`
   → 对应磁盘 `/var/media/movie.mp4`
   或 ==某个数据库记录==、==远端存储==，甚至是“上游拉流地址”。

也有的服务器不会用真实的 `.mp4` 扩展名，而是：

- `rtsp://example.com/live/room1` → live 源（例如来自上游推流）
- `rtsp://example.com/vod/12345` → 数据库里 id=12345 的某个 MP4

**所以扩展名只是“表象”**，具体怎么映射完全由服务器自己决定。

### 2. 打开 MP4：读取容器 + track 信息

很多实现会直接用 FFmpeg 的 libavformat（或类似库）：

大致步骤（伪代码级概念）：

```cpp
avformat_open_input(&fmt_ctx, "/var/media/movie.mp4", nullptr, nullptr);
avformat_find_stream_info(fmt_ctx, nullptr);

for each stream in fmt_ctx->streams:
    if stream->codecpar->codec_type == AVMEDIA_TYPE_VIDEO:
        // 读取 codec_id, width, height, extradata (SPS/PPS)
    if stream->codecpar->codec_type == AVMEDIA_TYPE_AUDIO:
        // 读取 codec_id, sample_rate, channels, extradata (ASC)
```

从这些信息中，你关心几个点：

- codec 类型：H.264 / H.265 / AAC / MP3 / …
- 视频宽高、Profile、Level、SPS/PPS
- 音频采样率、声道数、AudioSpecificConfig
- 容器里的 “extra data”（通常就包含构造 SDP 需要的字段）

### 3. 把“容器/编码元数据 → SDP 字段”

举两个典型的例子：

#### 3.1 H.264 视频 track → SDP

从 MP4 的 video track 中可以拿到：

- codec = H.264（4 字节代码 `avc1`）
- SPS / PPS（extradata，或从 bitstream 中解析）
- Profile/Level（在 SPS 里）

然后转换为：

- `m=video 0 RTP/AVP 96`
- `a=rtpmap:96 H264/90000`
- `a=fmtp:96 packetization-mode=1;profile-level-id=xxxxxx;sprop-parameter-sets=xxxx,yyyy`
  - `profile-level-id` 从 SPS 的 profile/level 拼出
  - `sprop-parameter-sets` 把 SPS/PPS 做 base64 编码
- `a=control:trackID=1`

#### 3.2 AAC 音频 track → SDP

从 MP4 音频 track 中拿到：

- codec = AAC
- sample rate = 44100
- channels = 2
- extradata = AudioSpecificConfig（两字节或更多）

然后转换为：

- `m=audio 0 RTP/AVP 97`
- `a=rtpmap:97 MPEG4-GENERIC/44100/2`
- `a=fmtp:97 ... config=<ASC 的十六进制>`
- `a=control:trackID=2`

> 也就是说：**容器/编码层的“extradata/SPS/PPS/ASC” → SDP 的 `fmtp/rtpmap` 行**。

### 4. 什么时候生成 SDP？

不同实现有不同做法，大致几种：

1. **第一次收到 DESCRIBE 时现生成 + 缓存**
   - 优点：简单，文件变了也能重新生成
   - 步骤：
     - 打开 MP4 → 解析 track → 生成 SDP 字符串
     - 缓存起来，后续 DESCRIBE 直接用
2. **服务启动时预扫描所有资源，离线生成 SDP**
   - 多用于资源列表固定的 VoD 系统（像 NVR、监控服务器）
   - 减少 DESCRIBE 时的 IO 开销
3. **对接上游/转码 pipeline**
   - 对于 live 输入，不是从 MP4 读，而是由上游模块（解复用/编码器）提供一个“媒体描述结构”，再统一转成 SDP。

## 三、结合 RTSP 具体流程看一下

把前面东西放回到你的 DESCRIBE 流程里：

> 客户端：DESCRIBE `rtsp://server:8554/movie.mp4`
>  服务器：解析 URI → 发现它对应 `/var/media/movie.mp4`

服务器做：

1. 找到媒体源（本地 MP4、上游拉流、数据库记录等）

2. 如果是 MP4：

   - 用容器库打开文件，得到 video/audio track 的编码信息
   - 每个 track 转成一个 `m=` 块 + 对应的 `a=rtpmap/a=fmtp/a=control`

3. 拼成完整 SDP：

   ```text
   v=0
   o=- 0 0 IN IP4 127.0.0.1
   s=movie.mp4
   c=IN IP4 0.0.0.0
   t=0 0
   a=control:*
   
   m=video 0 RTP/AVP 96
   a=rtpmap:96 H264/90000
   a=fmtp:96 ...
   a=control:trackID=1
   
   m=audio 0 RTP/AVP 97
   a=rtpmap:97 MPEG4-GENERIC/44100/2
   a=fmtp:97 ...
   a=control:trackID=2
   ```

4. 放到 DESCRIBE 响应 body 里，同时加上 header：

   ```text
   RTSP/1.0 200 OK
   CSeq: 2
   Content-Type: application/sdp
   Content-Length: <len>
   Content-Base: rtsp://server:8554/movie.mp4/
   
   <SDP 内容>
   ```

## 四、你现在实现 RTSP server 时可以怎么分阶段做？

结合你当前阶段，我建议你分两步：

### 第 1 阶段：**手写 SDP**，先把 DESCRIBE → SETUP → PLAY 流程打通

- 先不管 MP4 解析，写死一个 SDP：
  - 只做一个 video track（甚至不一定真发 H.264，可先发随便的数据）
  - 或手工写一个 H264 SDP，ffplay 也能连上
- 目标：
  - 解析 URI
  - 根据 URI 选择一个“内置 SDP 字符串”
  - 构造正确的 DESCRIBE 响应（Content-Type、Content-Length、Content-Base）

这一步可以让你完全熟悉：**RTSP 这一层的请求/响应 + SDP 字符串本身是什么**。

### 第 2 阶段：引入元数据解析（MP4 → SDP）

等 DESCRIBE 的流程稳定以后，再做“真实元数据”：

- 集成一个容器解析库（FFmpeg libavformat 比较常见）

- 封装一个函数，比如：

  ```cpp
  bool buildSdpFromFile(const std::string& filePath, std::string& outSdp);
  ```

  内部逻辑就是：打开 MP4 → 遍历 track → 生成 SDP。

- DESCRIBE 阶段就从“根据 URI 找到 filePath，再调用 `buildSdpFromFile`”变成动态 SDP。
