## 一、整体架构：net vs rtsp 两层的边界

你现在的划分：

- **net 模块**：管理连接、收发数据
- **rtsp 模块**：解析 RTSP 报文、生成 response、业务逻辑

这是非常合理的分层，关键是把两者的**边界**想清楚：

### 1. net 模块的“对上”接口

建议 net 层对上只暴露一种非常简单的抽象，比如逻辑上类似：

- 「有新连接」事件
- 「这个连接收到了若干字节数据」事件
- 「这个连接可以发送这些字节」的能力（比如一个 send 接口）

换句话说，rtsp 模块眼中看到的应该是：

- 一个“连接对象”（Connection / Channel 的概念）
- 连接对象上能：
  - 订阅/收到字节流
  - 发字节流
  - 知道自身的标识（fd / id）

**MVP 阶段 net 层可以简单粗暴：**

- 单线程 `accept` + `recv` + `send`，或者一个简单的 epoll 循环
- 不搞 reactor 模式、线程池、连接池这类高级玩法
- 不做太细的抽象，只要能把「bytes → 上层回调」和「上层给 bytes → send 回去」打通就足够

### 2. rtsp 模块的“对下”依赖

rtsp 模块应该**不感知** socket / epoll 等细节，只关心：

- 从某个“连接”上收到了完整或不完整的一段文本（RTSP 报文是一堆 ASCII 文本 + CRLF）
- 把生成的响应文本写回这个连接

所以 rtsp 层内部有理由自带一个「RTSP 连接对象」（比如 RtspConnection），它关联：

- 上层的 parser / handler / dispatcher 等
- 下层的 net 的 connection handle（一个抽象句柄，不必是裸 fd）

## 二、rtsp 模块内部：类职责的拆分建议

你提到的类：`handler / parser / dispatcher / builder / server`，我们可以给每个类一个非常清晰、**尽量单一**的职责。

可以先用一张表把 MVP 阶段的职责定死：

| 类                 | 主要职责（MVP 版本）                                         | 不做什么（刻意推迟的）                             |
| ------------------ | ------------------------------------------------------------ | -------------------------------------------------- |
| RtspServer         | 启动服务、注册到 net 模块；有新连接时创建 RtspConnection     | 不做业务逻辑、不解析报文、不直接操作 socket        |
| RtspConnection     | 维护单条 RTSP 连接上的解析状态：缓冲区、CSeq、当前是否在会话中等；驱动 parser / dispatcher / builder | 不做复杂 session 管理、不负责 RTP 发送             |
| Parser             | 把“字节/字符串”解析成 `RtspRequest` 对象（方法、URL、版本、头、body） | 不关心如何应答、不关心会话、不做任何网络 IO        |
| Builder            | 把 `RtspResponse` 对象序列化成字符串（状态行、header、空行、body） | 不决定状态码、不产生业务数据，仅负责“格式正确”     |
| Handler(接口+实现) | 根据 `RtspRequest` 生成 `RtspResponse` 的**业务决策**        | 不负责网络 IO、不负责解析、不负责底层序列化        |
| Dispatcher         | 根据 request 的方法（OPTIONS/DESCRIBE/SETUP/PLAY…）和 URL 选择对应的 Handler | 不直接处理业务、不做复杂路由（MVP 可仅按方法分发） |

下面逐个展开。

### 1. RtspServer：入口 + 与 net 绑定

**MVP 版本职责：**

- 在指定端口监听（通过 net 模块）
- 当 net 通知“有新连接”时：
  - 创建一个 `RtspConnection` 对象
  - 注册这个连接的“数据到达”回调（或者干脆在循环里手动调用 `onData()`）

关键思想是：**RtspServer 只负责“接生”和“登记”，不参与具体对话**。

未来扩展时，RtspServer 才会持有如：

- 全局 SessionManager
- 全局配置（认证开关、日志级别等）

但现在可以先不引入。

### 2. RtspConnection：一条 TCP 连接上的 RTSP 状态机

这是你 rtsp 模块里很重要但容易被忽略的类（你没写出来，但实际上需要它）。

它的职责是：

- 持有：
  - 底层 connection handle（从 net 来的）
  - 一个 buffer（字符串或字节数组）用来积累收到的数据
- 每次 net 层有数据时，调用：
  - `appendToBuffer()` 然后尝试从 buffer 中**切出一个完整的 RTSP 请求**（按空行分隔 headers，再按 Content-Length 截 body）
- 对每个完整请求：
  - 交给 `Parser` 解析为 `RtspRequest`
  - 交给 `Dispatcher` 找到合适的 `Handler`
  - 调用该 Handler，得到 `RtspResponse` 对象
  - 交给 `Builder` 序列化成文本
  - 通过 net 层发送出去

MVP 阶段 RtspConnection 暂时可以不维护：

- 复杂的 Session 表
- 多路 interleaved RTP 通道
- 超时时间、心跳等

只要保证：

- 一条 TCP 连接里，ffplay 发的多个 RTSP 请求你都能按顺序处理并回复

就符合“正确解析和返回报文”的 MVP 目标。

### 3. Parser：只负责「字符串 → RtspRequest」

建议把 parser 做成**尽量纯函数 / 无状态**，即：

- 输入：一段完整的 request 字符串（含首行 + headers + 空行 + 可能的 body）
- 输出：一个简单的 `RtspRequest` 结构体，包含：
  - 方法（字符串/枚举）
  - URL
  - 版本
  - 头部字段 map
  - body 文本（MVP 阶段可能只有 SDP 或为空）

为了不超前设计，先刻意不考虑：

- pipeline 模式下，多条请求黏在一起（这个由 RtspConnection 的 buffer 分片来承担）
- 错误恢复策略（解析失败可以先直接返回 400 Bad Request）

也就是说，parser 可以假设：

> 我拿到的是“一条 RTSP 请求的完整文本”。

至于“怎样切出来这条完整请求”，是 RtspConnection 的责任。

### 4. Builder：只负责「RtspResponse → 字符串」

同样尽量无状态，职责是：

- 把 `RtspResponse` 里的：

  - 状态行（版本、状态码、reason phrase）
  - header map
  - body

- 按 RTSP 规范的格式拼成：

  ```
  RTSP/1.0 200 OK\r\n
  CSeq: 1\r\n
  ...
  \r\n
  body...
  ```

MVP 阶段可以硬编码一些简单规则：

- 总是输出 `RTSP/1.0`
- 默认必带 `CSeq`、`Date`，其余 header 按需要拼

Builder 不需要懂“业务”，只要负责**格式正确**，提高报文可靠性。

### 5. Handler：接口 + 各 Method 的实现

你已经有正确的想法：先定义一个统一的 Handler 接口（抽象类），比如逻辑上类似：

- 输入：`RtspRequest` +（可能的上下文，比如 RtspConnection / Server-level info）
- 输出：`RtspResponse`

然后针对不同类型 request 定义不同 Handler：

- `OptionsHandler`
- `DescribeHandler`
- `SetupHandler`
- `PlayHandler`
- （MVP 阶段甚至可以先支持 `OPTIONS` + `DESCRIBE` 就足够让 ffplay 建立初始交互）

MVP 的时候 Handler 内部可以很“土”：

- `OptionsHandler` 直接返回固定的 `Public: OPTIONS, DESCRIBE, SETUP, PLAY, TEARDOWN` 等 header
- `DescribeHandler` 返回一个固定的 SDP 文本（模拟一个单路 H264 视频流）
- `SetupHandler` 和 `PlayHandler` 甚至可以先返回“看起来合理的 200 OK”，而不真正发送 RTP，让 ffplay 至少走完整个 RTSP 流程

**关键点：**

- Handler 做的是“**决策**”：根据 request 选择状态码 / header / body
- 解析和序列化完全交给 Parser / Builder
- 网络收发由 RtspConnection / net 层负责

### 6. Dispatcher：请求到 Handler 的路由器

Dispatcher 的任务就是：从 `RtspRequest` 的特征中选择哪个 Handler 来处理。

MVP 阶段可以非常简单：

- 只按方法（`OPTIONS`, `DESCRIBE`, `SETUP`, `PLAY`…）分发
- 不区分 URL、路径、流名等

未来想扩展的时候，可以改变成：

- 按 method + URL 路由（比如 `/live/1` 用一个 handler，`/vod/xxx` 用另一个 handler）
- 或者支持注册自定义 handler（类似 HTTP 框架的路由表）

但现在完全不需要，甚至可以硬编码一个 `switch(method)`。

## 三、Server 类的定位

你写的 `server` “负责启动”这个定位很好，但建议你在概念上区分两个东西：

1. **Process 级别的 Server（Main 入口那层）**
   - 负责解析命令行参数（监听端口、日志级别）
   - 创建 net 模块的主 loop（或 event base）
   - 创建 RtspServer 实例，并把两者挂起来
   - 进入事件循环
2. **RTSP 逻辑层的 RtspServer 类**
   - 负责和 net 交互：监听新连接、创建 RtspConnection
   - 持有 dispatcher / handler 集合（如果你决定放这里）
   - 也可以持有一些全局配置结构体

MVP 阶段可以先不严格切分这两个概念，但心里要清楚：

- main.cpp 那层负责“进程级的事”
- RtspServer 是“协议级的 server 对象”

## 四、如何避免过度设计（结合当前 MVP）

你的目标非常明确：

> 正确解析和返回 RTSP 报文，让宿主机上的 ffplay 能正常与 server 完整交互。

在这个目标下，可以有意识地**不做**的东西包括：

1. **不做高性能网络模型**
   - 单线程 + 阻塞或简单的 epoll 就够用
   - 不做线程池、不做多进程、不搞锁优化
2. **不做完整的 RTP 推流**
   - 可以暂时不发送真正的 RTP 包，只在控制面上完成 OPTIONS / DESCRIBE / SETUP / PLAY 流程
   - 或者只实现最简单的 UDP RTP 推流，单个客户端，码流可以是录制好的固定帧
3. **不做复杂 Session 管理**
   - Session-ID 表、超时回收、多个客户端对同一个流等，可以统统延后
   - MVP 可以：“只要 ffplay 能拉一条流，退出时资源不会立刻崩，就行”
4. **不做插件化 / 复杂配置系统**
   - Handler 的注册方式可以先 hardcode
   - SDP 固定写死在代码里没问题（后期再抽出来）

反过来说，你现在应该重点打磨三件事：

1. **报文切分 + 解析稳定性**
   - 解决黏包、半包的问题
   - 任何合法的单条 RTSP 请求都要能正确解析
2. **response 的格式完全合法**
   - 状态行、CSeq 对应、必要 header、CRLF 格式正确
   - ffplay 对 RTSP 很敏感，细节错了会直接失败
3. **连接生命周期清晰**
   - ffplay 正常退出或者 server 主动关闭时，资源不会泄露、不会崩

## 五、一个典型交互流程（从 net 到 rtsp）

以 ffplay 为客户端的典型流程，逻辑上可以想象成：

1. net 接受新连接 → 通知 RtspServer
2. RtspServer 创建 RtspConnection，注册到 net
3. ffplay 发送 `OPTIONS`：
   - net 收到字节 → 交给 RtspConnection
   - RtspConnection 把字节放入 buffer，并检测到形成一个完整的请求
   - 调用 Parser → 得到 RtspRequest
   - 调用 Dispatcher → 找到 OptionsHandler
   - 调用 OptionsHandler → 得到 RtspResponse（200 + Public 等头）
   - Builder 序列化 → 字符串
   - RtspConnection 调 net 的 send 接口发送
4. 后续 DESCRIBE / SETUP / PLAY 流程类似

只要你把上面这个“流水线”在代码结构层面梳理清楚，哪怕内部实现还很粗糙，MVP 就已经成立。
