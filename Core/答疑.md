# 📘 第一部分

# **一本书（深度）——到底要读哪些？为什么是它们？**

你不是要“多读”，而是要“读对”。
 深度型书籍的目标是——**构建底层原理、建立世界观、打基础的基础。**

对于你的方向（C++ + 系统 + 网络 + 实时音视频），下面是强烈推荐、经过行业验证的清单：

------

## **📚（A）系统底层 + 程序运行原理**

### **1）CS:APP《深入理解计算机系统》**

**价值：**
 最好的 computing 底层书籍，没有之一。
 你现在所有的痛点（内存、字节序、网络 IO、性能）背后都是它的内容。

你能从中获得：

- CPU 如何处理指令
- 地址空间是怎样的
- 字节序为什么存在
- 系统调用如何工作
- 缓冲、IO、本质是什么

📌 **这本书会让你“看清整个计算机”。**

------

## **2）《UNIX 网络编程》（UNP）**

**价值：**
 这是你做网络工程（epoll、socket、RTSP/RTP）最核心的底层知识。

你能从中获得：

- socket API 原理
- TCP/UDP 的真实工作方式
- IO 多路复用（epoll）
- 超时、缓冲、阻塞、半关闭
- 实战场景的真实设计思路

📌 **你的 RTSP Server 的一半知识都来自这本书。**

------

## **3）《Linux 高性能服务器编程》（游双）**

**价值：**
 UNP 偏原理，这本书偏工程。
 你能快速掌握“如何写一个高性能 server”。

内容包括：

- epoll reactor 模型
- 多线程、任务队列
- 连接池
- HTTP 服务器框架

📌 **这本书相当于你未来要做的 Server 的工程蓝本。**

------

## **📚（B）网络协议 + Multimedia**

### **4）《TCP/IP 卷一》**

内容非常扎实，适合真正理解协议细节：

- TCP 握手、流控、拥塞控制
- UDP、IP 分片
- 端口、头部结构

📌 为什么你的 RTP 会出问题？
 大多都能在这本书找到原理。

------

### **5）RFC 文档**

你做 RTSP/RTP 一定要看：

- RFC 2326（RTSP）
- RFC 3550（RTP）
- RFC 6184（H264 over RTP）

📌 “搭积木”必须看说明书。

------

## **📚（C）C++ 工程能力（必选其一）**

### **6）《Effective C++ / Modern C++ 系列》**

教你真正的“工程 C++”，而不是语法。

你能提升：

- RAII
- 模板基础
- 指针、引用使用规约
- 代码安全性、性能
- 模块化习惯

------

## **📚 总结：你真正需要的 5 本书**

| 领域       |        书名        | 价值                         |
| ---------- | :----------------: | ---------------------------- |
| 系统底层   |       CSAPP        | 构建整个计算机世界的底层认知 |
| 网络原理   |    TCP/IP 卷一     | 深入协议本质                 |
| 网络工程   |        UNP         | 系统理解 socket、IO、epoll   |
| 服务器工程 | Linux 高性能服务器 | 最快工程落地路线             |
| C++ 工程   |   Effective C++    | 让代码能跑、能维护           |

你不需要十几本书。
 **这五本就足够你做到“能独立完成 RTSP/系统项目”。**

# 🧩 第三部分

# **开源代码（工程）——为什么必须？要学什么？怎么用？**

你现在不理解开源项目的意义，我来讲得非常透彻：

------

# ✔（A）为什么你必须看开源代码？

理由只有一个：
 **开源项目 = 工程真相。**

你要学：

- 模块如何拆分
- 整体架构如何设计
- 状态机如何实现
- 如何处理错误
- 如何处理协议细节
- 如何管理线程/缓存/内存
- 如何组织目录结构

这些东西 **课程不教、书不讲、教程不覆写**，
 但开源项目里全都有。

它是“真实世界的样子”。

你现在做 RTSP Server，必须知道行业内的代码怎么组织。

------

# ✔（B）要学什么？（关注点不是读懂，而是提取“工程设计模式”）

建议你从下面学习点切入：

### **1）整体架构是怎么拆解的？**

例如：

```
server
 ├── session
 ├── transport
 ├── parser
 ├── scheduler
 ├── rtp
 └── utils
```

你会看到 **模块化能力** 是如何落地的。

------

### **2）每个模块的职责是什么？**

例如 live555：

- RTSPServer
- RTSPClientSession
- OnDemandServerMediaSubsession
- FramedSource
- RTPSink

你会理解工程中的“责任边界”。

------

### **3）状态机怎么写？**

RTSP 是典型状态机。

看开源 = 看“行业标准写法”。

------

### **4）错误处理怎么写？**

工程不是“正确场景”，工程是“错误场景”。

你要学：

- 超时怎么办
- fd 损坏怎么办
- 连接断开怎么办
- UDP 缓冲区满怎么办

这些写法都在开源里。

------

### **5）怎样处理 RTP 构包？**

你要学：

- 时间戳
- sequence
- payload
- 分片

这些你写不完一个人在 RFC 里硬啃会非常难。

------

### **6）怎样管理 buffer？**

例如使用：

- ring buffer
- frame queue
- packet queue

这是你写高性能系统的关键。

------

### **7）线程模型怎么组织？**

例如：

- 单线程
- reactor
- producer/consumer
- event loop

这都是你工程写不出来但开源现成的思想。

------

# ✔（C）怎样用在你自己身上？

非常简单：
 你不是“读懂开源代码”，你是“抽象开源项目的设计”。

流程如下：

------

## **① 明确你自己项目的结构**

然后对照开源项目：

- 目录结构
- 模块列表
- 类职责
- 线程模型

例如：
 你写 session 类时，就看看 live555 的 session 模块长什么样。

------

## **② 借鉴模块拆分方式，而不是复制代码**

例如：

- Parser 模块
- Session 模块
- Transport 模块
- MediaSource 模块
- Scheduler 模块

你借鉴“拆法”，而不是“实现细节”。

------

## **③ 遇到瓶颈就去开源项目找答案**

例如你不知道：

- NALU 分片
- RTP 头字段
- UDP 缓冲策略
- 发送时序
- frame → packet 的处理方式

开源项目里都有标准解法。

------

## **④ 用开源项目检查自己的“盲区”**

例如你发现：

“我的 session 类没处理 teardown”
 “我没处理 NAT 环境的端口问题”
 “我没有 jitter buffer”

开源项目会告诉你自己漏了什么。

------

## **⑤ 最后，将“模式”内化成自己的设计习惯**

这就是“工程化水平提升”。
