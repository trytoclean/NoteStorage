异步非阻塞，eg

```c++
wait_event();
  
swtich(event){
case 1:accept();
case 2:data_arrive();
case 3:deal();

case 5:on_error();

```

RPC remote procedure call 远程程序调用

​	是一种基于请求响应模型的机制，如果按照字面意思，就是用于处理的函数在另一台主机处理，即处理请求和处理业务的逻辑分开，那么是这样的吗，总之RPC会带来哪些好处？



任务/数据分解+请求/响应队列 vs 时间循环+状态机

首先说明两者的工作机制：

前者，是RPC的形式，也就是请求响应模型。任务的响应和业务分开处理；通过添加队列，队列起到一个缓冲的作用。请求之后放到请求队里中，响应之后放到响应队列中去，

使用一个IO线程处理请求，使用线程池同步/异步处理业务 （worker 线程）

弊端，多次拷贝，请求一次，处理一次，响应一次，发送一次

后者，事件循环，我认为是一整套业务流程（或者说一次请求响应的过程），状态机，是描述当前当个请求的状态。二者联系不知道。

上述二者的差异：RPC vs 流媒体处理

RPC：主要可用作文件服务器，请求数量不多，但是单次请求带宽大。后者适合音视频场景下，使用纯异步非阻塞，减少内存拷贝。
