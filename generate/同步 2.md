# 一、总体原则：基础 ≠ API 记忆

在并发领域，“基础”不是：

- 会用 `std::thread`
- 记住 `mutex / cv / atomic` 的接口

**而是你脑子里要有这三张图：**

1. **线程是如何被调度的**
2. **共享内存是如何被破坏的**
3. **同步原语是如何恢复秩序的**

如果你学完后，能在纸上“推演错误是如何发生的”，说明基础扎实。

------

# 二、第一层：线程与执行模型（Execution Model）

## 1. 必须建立的底层认知

### 线程是什么（操作系统视角）

- 线程是 **CPU 调度的最小单位**
- 一个进程内：
  - 多个线程
  - 共享地址空间
  - 独立执行流（PC + 寄存器）

你要形成这样一个**心智模型**：

```
进程地址空间
 ├─ 全局区
 ├─ 堆
 └─ 栈
      ├─ 线程 A 栈
      └─ 线程 B 栈
```

------

## 2. 学习顺序（非常克制）

只做三件事：

1. 创建线程
2. 等待线程
3. 理解生命周期

```cpp
std::thread t([] {
    // 并发执行
});
t.join();
```

### 此阶段禁止做的事

- ❌ mutex
- ❌ atomic
- ❌ condition_variable

你要先**看清问题**，再学“解决方案”。

------

## 3. 必须回答的问题

- 为什么下面的代码是**未定义行为**？

```cpp
int x = 0;
std::thread t([&] {
    x = 42;
});
```

- 为什么线程函数不应捕获局部变量引用？
- `join()` 为什么不是可选项？

------

# 三、第二层：数据竞争（Data Race）——并发的原罪

这是整个并发体系的**原点概念**。

## 1. 什么是数据竞争（标准定义）

> 两个或以上线程：
>
> - 同时访问同一内存
> - 至少一个是写
> - 没有同步

结论：**行为未定义**

------

## 2. 必须亲眼看到的问题

```cpp
int x = 0;

void f() {
    for (int i = 0; i < 100000; ++i)
        ++x;
}
```

你需要反复运行：

- 单线程：结果稳定
- 多线程：结果随机

并问自己：

> 为什么“++”不是原子操作？

------

## 3. 这一层的验收标准

你能：

- 用“读-改-写”拆解任意一行代码
- 判断一段代码是否存在数据竞争
- 明确区分：
  - race condition（逻辑问题）
  - data race（内存模型问题）

# 四、第三层：互斥与临界区（Mutex & Critical Section）

现在你已经**看清楚问题**了，才有资格学 mutex。

------

## 1. mutex 的本质（不是锁）

> mutex 是一种**执行顺序约束**

它做的事只有一件：

> 在时间轴上，把并发访问“串行化”

------

## 2. 正确学习姿势

### 只学这三样

- `std::mutex`
- `std::lock_guard`
- 临界区边界

```cpp
{
    std::lock_guard<std::mutex> lk(m);
    // 临界区
}
```

### 必须理解的“感觉”

- 锁不是保护代码
- **锁保护的是数据**

------

## 3. 思维训练（非常重要）

看到下面代码，你要**下意识不安**：

```cpp
if (x > 0) {
    std::lock_guard<std::mutex> lk(m);
    --x;
}
```

因为：

- 判断与修改不在同一临界区

------

## 4. 验收标准

你能：

- 正确划定临界区边界
- 说清楚“为什么锁太大/太小都不对”
- 解释什么是死锁（无需解决）

------

# 五、第四层：条件变量（Condition Variable）——等待的正确方式

这是**并发中最容易被误用**的工具。

------

## 1. 条件变量解决什么问题？

> **线程如何高效地等待某个状态发生变化**

不是 sleep，不是轮询。

------

## 2. 唯一正确的使用模型

```cpp
std::unique_lock<std::mutex> lk(m);
cv.wait(lk, [] {
    return condition;
});
```

你要理解：

- 为什么必须配合 mutex
- 为什么条件不是 cv 本身
- 为什么会虚假唤醒

------

## 3. 必须形成的认知

> 条件变量是“状态变化通知机制”，不是“事件”

------

## 4. 验收标准

你能：

- 写出标准等待模式
- 解释虚假唤醒
- 判断 `notify_one` vs `notify_all`

------

# 六、第五层：原子操作（Atomic）——最小同步单位

这是**高级基础**。

------

## 1. 原子操作的定位

- 不是 mutex 的替代
- 是性能优化工具

你现在只需要理解：

```cpp
std::atomic<int> x{0};
++x;
```

------

## 2. 你必须知道的边界

- 原子 ≠ 无锁万能
- 原子只能保证：
  - 单变量一致性

------

## 3. 验收标准

你能说清楚：

- 为什么 atomic 不能保护复杂结构
- 为什么“原子 + 逻辑”仍然可能错

------

# 七、最后一层：进程 & IPC（只做概念对照）

这里不深入 API，只建立**对照认知**。

| 对比项   | 线程 | 进程 |
| -------- | ---- | ---- |
| 地址空间 | 共享 | 隔离 |
| 同步需求 | 必须 | 较少 |
| 崩溃影响 | 全部 | 局部 |

你要得出一个结论：

> 并发设计的第一原则：
> **能不用共享，就不用共享**

------

# 八、你的“基础是否扎实”的最终判据

你可以用下面 5 个问题自测：

1. 我能否判断一段代码是否存在 data race？
2. 我能否画出多个线程对同一变量的时间线？
3. 我是否清楚每一个 mutex 在保护什么？
4. 我是否知道 condition_variable 在等“什么”？
5. 我是否知道什么时候不该用 atomic？

如果全部是“是”，你已经超过了**绝大多数只会用 API 的人**。
