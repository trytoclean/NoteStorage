**功能需求（Functional Requirements）**
指系统**必须做什么**，即系统对外提供的具体功能与行为，直接面向用户或其他系统，描述“**能不能完成某件事**”。

- 关注点：业务能力、功能正确性
- 典型问题：系统支持哪些操作？流程是什么？输入输出是什么？
- 示例：
  - 支持 RTSP 的 `OPTIONS / DESCRIBE / PLAY` 请求
  - 客户端可以推流 RTMP，服务端能转发给多个播放端
  - 用户通过 HTTP 接口拉取 `.flv` 直播流

一句话概括：**功能需求 = 系统要提供哪些功能。**

------

**非功能需求（Non-Functional Requirements, NFR）**
指系统在实现功能时**要做到什么程度**，描述功能之外的质量属性和约束，回答“**做得好不好、稳不稳、快不快**”。

- 关注点：性能、并发、稳定性、可用性、可维护性、可部署性等
- 典型问题：能承受多少并发？多快？多稳定？怎么部署和观测？
- 示例：
  - 单机支持 10 万 TCP 长连接
  - RTSP 播放延迟 P95 小于 200 ms
  - 服务可通过 Docker 部署，支持优雅退出
  - 运行 24 小时内无内存泄漏

一句话概括：**非功能需求 = 功能的质量、性能和约束条件。**

------

**对比总结**

| 维度       | 功能需求         | 非功能需求               |
| ---------- | ---------------- | ------------------------ |
| 回答的问题 | 做什么           | 做到什么程度             |
| 面向对象   | 用户 / 业务      | 系统 / 架构              |
| 是否可见   | 通常可见         | 通常不可见               |
| 失败表现   | 功能不可用       | 慢、不稳、扛不住         |
| 示例关键词 | 支持、提供、实现 | 延迟、并发、稳定、可观测 |

------

**工程视角的核心结论**

- **没有功能需求，系统没用**
- **没有非功能需求，系统不可用、不可扩展、不可交付**

在你的流媒体项目中：

- *协议支持、推拉流能力* 是 **功能需求**
- *高并发、弱网适应、性能指标、Docker 部署* 是 **非功能需求**
