GTest（GoogleTest）几乎可以测试 **你 RTSP 项目中的所有部分**，只要这些部分可以用 C++ 调用、构造、执行。
 下面我从**项目实际场景**出发，把可测试内容分成 8 大类，让你能完全知道哪些地方应该写测试、哪些地方不适合写测试。

------

# ✅ 1. **纯函数逻辑测试**（最基本，也最稳定）

适合你 RTSP 项目中的：

- `string_to_method()`
- `parse_request_line()`
- `build_response()`
- `enum_to_string()`
- `RtspResponse::serialize()`
- RTP Header 组包函数
- SDP 行构建函数
- URL/URI 解析

示例：

```cpp
TEST(ParserTest, ParseMethod) {
    EXPECT_EQ(string_to_method("OPTIONS"), Method::OPTIONS);
}
```

**适用范围：任何没有副作用的逻辑。**

------

# ✅ 2. **类的行为测试 / 成员函数测试**

你现在已经有：

- `RtspParser`
- `RtspDispatcher`
- `ResponseBuilder`
- `IRtspHandler + XxxHandler`

这些都可以单独测试。

示例：

```cpp
TEST(DispatcherTest, OptionsReturns200) {
    RtspDispatcher d;
    RtspRequest req;
    req.method = Method::OPTIONS;
    req.headers["CSeq"] = "1";

    RtspResponse res = d.dispatch(req);

    EXPECT_EQ(res.code, 200);
}
```

这种测试可以确保某个类逻辑正确，不依赖网络。

------

# ✅ 3. **模块之间的交互测试（不 run 整个 server）**

例如：

- Parser → Dispatcher 的组合行为
- Dispatcher → Handler → ResponseBuilder 的链路
- Request 结构体 → 处理后 → Response 结构体

即“单元 + 模块级联”测试。

示例：

```cpp
TEST(IntegrationTest, ParseAndDispatch_OPTIONS) {
    std::string data = "OPTIONS rtsp://test RTSP/1.0\r\nCSeq: 1\r\n\r\n";

    RtspParser parser;
    auto req = parser.parse(data);

    RtspDispatcher dispatcher;
    auto res = dispatcher.dispatch(req);

    EXPECT_EQ(res.code, 200);
}
```

这种测试帮助你模拟 ffplay 的行为，而不用真正建立 socket。

------

# ✅ 4. **异常与错误情况的测试**

例如：

- 不合法 method
- 缺失 CSeq
- 不合法请求行
- Handler 不存在
- Parser 遇到不完整内容
- Session 不存在时报错

示例：

```cpp
TEST(ParserTest, InvalidRequestLineThrows) {
    RtspParser parser;
    EXPECT_THROW(parser.parse("INVALID"), std::runtime_error);
}
```

在协议类项目里，**错误测试非常重要**。

------

# ✅ 5. **行为边界测试（input boundary tests）**

举例：

- 超长 Header
- 空 Body
- 未知 Header
- method 小写、大写、混合大小写
- RTP seq 溢出
- 时间戳 wrap-around

RTSP/RTP 很多问题都来自边界情况，GTest 非常适合这种细节的验证。

------

# ✅ 6. **多线程/并发相关的逻辑（有限度）**

例如：

- ThreadPool 的任务提交
- ConnectionManager 的 add/remove
- Session 管理（session timeout）
- 环形缓冲区（RingBuffer）

⚠️ 注意：
 GTest 不适合做高负载并发测试，只适合检查逻辑正确性。

示例：

```cpp
TEST(ThreadPoolTest, TaskExecuted) {
    ThreadPool pool(4);
    std::atomic<int> counter{0};

    pool.enqueue([&](){ counter++; });
    std::this_thread::sleep_for(std::chrono::milliseconds(10));

    EXPECT_EQ(counter.load(), 1);
}
```

------

# ❌ 7. 不适合用 GTest 的部分

这也很重要：

## ❌（1）整个 RTSP 服务器的网络功能（集成测试更合适）

比如：

- 启动 server
- ffplay 连接
- 收 RTP 包

这种规模建议用：

- Python script
- curl / ffprobe
- 自己写的小客户端
- Docker 环境跑 e2e 测试

GTest 可以做，但不推荐。

------

## ❌（2）性能测试（要用 benchmark）

GTest 不擅长性能，应该用 Google Benchmark。

------

## ❌（3）大规模异步 IO（完整 epoll 事件流）

比如：

- epoll/accept 的完整流程
- socket 的 read/write

这些更适合使用专用工具或 integration test。

------

# ✅ 8. GTest + Mock（可以模拟 socket / handler）

使用 GoogleMock（GTest 自带）可以 mock：

- Fake Socket
- Fake Connection
- Fake Parser
- Fake Handler

例如模拟 SETUP/PLAY 请求链路，无需网络。

示例：

```cpp
class MockHandler : public IRtspHandler {
public:
    MOCK_METHOD(RtspResponse, handle, (const RtspRequest&), (override));
};
```

这对构建一个大型的 RTSP 框架很实用。

# 🎯 你现在的 RTSP 项目最应该写的测试（按优先级）

我按你的项目进度给优先级：

### **优先级 A（必须）**

- RtspParser（请求行、header、方法）
- Dispatcher（匹配正确 handler）
- ResponseBuilder（序列化、状态码）
- Utilities（字符串切分、大小写转换、enum 转换）

### **优先级 B（重要）**

- SDP 生成
- RTP header pack/unpack
- URI 解析

### **优先级 C（稍后）**

- Session 管理
- ThreadPool（你问过 util 放哪里，这个也该测试）
- ConnectionManager

### **优先级 D（未来）**

- 缓存（ring buffer）
- media source（mp4/h264 的 frame reader）
