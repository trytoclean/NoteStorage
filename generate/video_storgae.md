非常好，这个问题非常关键：

> “视频文件是怎样被实际存储的？容器格式（MP4、MKV、TS）到底在存什么？它们的区别又是什么？”

这涉及**容器格式（Container Format）**、**编解码格式（Codec）**、**封装结构（Muxing Structure）** 三层知识。
 我们一层一层来梳理。

------

## 🧱 一、基本概念层次

| 层次             | 名称      | 作用                                            | 举例                        |
| ---------------- | --------- | ----------------------------------------------- | --------------------------- |
| **原始媒体数据** | Raw       | 原始像素或采样点                                | YUV、PCM                    |
| **编码格式**     | Codec     | 压缩算法（减小体积）                            | H.264、H.265、AAC、Opus     |
| **容器格式**     | Container | 把多个轨道（音/视频/字幕/元数据）打包成一个文件 | MP4、MKV、TS、AVI、MOV、FLV |

### 简单比喻：

> - **Codec** 是压缩算法（像 ZIP）。
> - **Container** 是打包盒子（像 ZIP 文件本身）。
> - 你可以用 H.264 + AAC 放进 MP4，也可以放进 MKV；只是“盒子”不同。

------

## 🧩 二、视频文件在磁盘上的实际结构

以 **MP4 文件** 为例（它是最常见的 ISO Base Media File Format 派生格式）：

```
+--------------------------------------------------------+
| ftyp | moov | mdat | free | udta | ...                 |
+--------------------------------------------------------+
```

### 各部分解释：

| Box           | 名称           | 功能                                                   |
| ------------- | -------------- | ------------------------------------------------------ |
| **ftyp**      | File Type Box  | 说明文件类型和兼容性（如 mp42, isom）                  |
| **moov**      | Movie Box      | 元数据（metadata）部分：时长、轨道信息、索引表、时间戳 |
| **mdat**      | Media Data Box | 实际存放音视频帧数据（H.264 NALU、AAC帧）              |
| **udta**      | User Data Box  | 可选的自定义元数据（作者、标题等）                     |
| **free/skip** | 空白或占位块   | 用于快速编辑或对齐                                     |

👉 实际磁盘存储顺序中：

- **音视频帧是交织存储的**（interleaved）
- 每个帧都在 `mdat` 里；
- `moov` 保存偏移表（告诉播放器帧在文件中的位置）。

------

## 🎞️ 三、容器中保存的核心数据类型

以 H.264 + AAC 轨道为例：

| 类型                     | 示例内容                   | 说明                               |
| ------------------------ | -------------------------- | ---------------------------------- |
| **视频帧（H.264 NALU）** | SPS / PPS / IDR / P / B 帧 | 记录压缩后的图像帧数据             |
| **音频帧（AAC）**        | ADTS 帧                    | 每帧包含固定采样数（如 1024 样本） |
| **时间戳**               | PTS/DTS                    | 播放与解码时序                     |
| **索引表**               | stco/stsz/stts             | 快速定位帧在文件中的偏移和大小     |
| **元数据**               | title, creation_time       | 辅助信息                           |

🧩 意义：

> 一个视频文件，其实就是**音频轨 + 视频轨 + 时间索引 + 元数据的集合体**。

------

## 🧠 四、不同容器格式的差异

| 容器格式         | 特点                     | 优点                     | 缺点              | 典型场景         |
| ---------------- | ------------------------ | ------------------------ | ----------------- | ---------------- |
| **MP4**          | ISO标准，块状结构（Box） | 通用、兼容性强、流式支持 | 编辑困难          | 在线播放、移动端 |
| **MKV**          | 开源、灵活               | 支持任意编码、多字幕     | 不被所有设备支持  | 高质量本地存储   |
| **TS (MPEG-TS)** | 188字节包结构            | 容错高、可流化           | 冗余大            | 直播传输、广播   |
| **MOV**          | 苹果QuickTime            | 支持专业元数据           | 较重              | 专业后期制作     |
| **FLV**          | Flash容器                | 简单、轻量               | 已过时            | 早期网页流媒体   |
| **AVI**          | 早期微软格式             | 简单                     | 不支持B帧、无索引 | 老视频、兼容场景 |

📦 **总结：**

> MP4适合存储 + 点播，TS适合直播传输，MKV适合本地存档。

------

## ⚙️ 五、文件存储的物理特征

| 特征                         | 含义                                       | 示例          |
| ---------------------------- | ------------------------------------------ | ------------- |
| **交错存储 (Interleaved)**   | 音频帧和视频帧交替排列，播放时无需频繁seek | MP4、MOV      |
| **非交错 (Non-interleaved)** | 视频集中存储，音频集中存储                 | 原始封装格式  |
| **索引表 (Index Table)**     | 文件头保存偏移表，用于随机访问             | MP4的`stco`表 |
| **块对齐 (Box Alignment)**   | 各Box以4字节或8字节对齐                    | 提高I/O效率   |

------

## 💡 六、文件到流媒体的关系

| 层次          | 文件内角色   | 流媒体角色  |
| ------------- | ------------ | ----------- |
| H.264 NALU    | 视频帧内容   | RTP Payload |
| AAC 帧        | 音频内容     | RTP Payload |
| moov + 索引表 | 元数据       | SDP描述     |
| mdat          | 实际媒体数据 | RTP流内容   |

👉 也就是说，**RTP发送的内容就是容器中 mdat 里的压缩帧序列，只是换了一种打包方式。**

------

## 🧭 七、延伸思考：视频服务器存储结构

在大规模视频网站（如 Bilibili、YouTube）上，存储结构会这样演变：

| 层次     | 存储形式                       | 说明                       |
| -------- | ------------------------------ | -------------------------- |
| 原始文件 | `.mov`、`.mp4`                 | 上传后临时存储             |
| 转码结果 | `.mp4`（多码率、多分辨率）     | 用 FFmpeg 转码生成多个版本 |
| 分片文件 | `.ts` 或 `.fmp4`               | 用于 HLS/DASH 分发         |
| 元数据   | 数据库记录（时长、格式、封面） | 支持检索与播放控制         |
| 缓存/CDN | 小片段缓存                     | 提升并发和加载速度         |

------

## 🔍 总结一句话

> 视频文件不是“单一路径的视频流”，
>  而是一个**多轨封装容器**，
>  存放了音频轨、视频轨、时间信息、索引表等多种数据，
>  编码（Codec）负责压缩，容器（Container）负责组织，
>  而流媒体（RTSP/HLS/DASH）只是换了一种传输方式来读这些数据。
