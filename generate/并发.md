## 一、先建立整体认知：你在解决什么问题？

在工程视角下，这四个概念并不是并列的，而是**分层出现的工具**：

```
操作系统
 ├─ 进程（资源隔离单位）
 │   ├─ 线程（CPU 调度单位）
 │   │   ├─ 并发执行
 │   │   └─ 线程同步（保证正确性）
 │   └─ 进程间通信（打破隔离）
```

**核心问题只有两个：**

1. **并发执行带来性能**
2. **并发执行破坏正确性**

C++ 多线程学习，本质就是在 **“性能” 与 “正确性”** 之间建立可控的工程平衡。

------

## 二、第一阶段：线程（Thread）——理解“并发执行”的最小单位

### 1. 学什么（只学必要的）

- `std::thread`
  - 创建 / join / detach
- 线程函数的三种形式
  - 普通函数
  - lambda
  - 成员函数
- 线程生命周期

### 2. 核心认知（非常重要）

- **线程 ≠ 并行**
  - 并发：逻辑上同时
  - 并行：物理上同时（多核）
- **线程共享进程内所有资源**
  - 地址空间
  - 堆
  - 全局变量
  - 文件描述符

> 这正是“危险”的根源。

### 3. 验收标准

你应当能回答：

- 为什么线程创建“很便宜”，但依然不能乱用？
- `detach()` 的工程风险是什么？
- 为什么线程函数不能轻易捕获局部变量引用？

------

## 三、第二阶段：线程同步——并发的“正确性约束”

这是 **C++ 多线程的核心难点**，也是多数初学者翻车的地方。

### 1. 同步解决的本质问题

> **多个线程同时访问共享状态，如何避免不一致？**

这在工程上分为三类问题：

| 问题类型   | 表现                 |
| ---------- | -------------------- |
| 数据竞争   | 随机崩溃、错误结果   |
| 原子性破坏 | 中间态被看到         |
| 可见性问题 | 修改未被其他线程看到 |

------

### 2. 学习顺序（强烈建议）

#### （1）互斥量：`std::mutex`

- `lock / unlock`
- `std::lock_guard`
- `std::unique_lock`

**核心思想：**

> 用“临界区”把共享数据包起来

```cpp
std::mutex m;
int counter = 0;

void f() {
    std::lock_guard<std::mutex> lk(m);
    ++counter;
}
```

你要学会**看到共享变量就条件反射地问一句**：

> 谁在保护你？

------

#### （2）条件变量：`std::condition_variable`

解决的问题不是互斥，而是：

> **线程之间如何“等待某个条件成立”？**

典型模式：

```cpp
while (!condition) {
    cv.wait(lock);
}
```

你需要理解：

- 为什么一定是 `while` 而不是 `if`
- `wait` 为什么会“虚假唤醒”
- 条件变量 ≠ 事件通知

------

#### （3）原子操作：`std::atomic`

这是**性能与正确性的折中方案**。

- 适合：
  - 计数器
  - 状态位
- 不适合：
  - 复杂不变量
  - 多字段一致性

你必须理解：

- 原子 ≠ 无锁万能
- 内存序（至少知道存在）

------

### 3. 验收标准

你应当能：

- 写出一个**正确的生产者-消费者模型**
- 解释死锁产生的 4 个必要条件
- 知道“锁粒度”如何影响性能

## 四、第三阶段：进程（Process）——资源隔离与系统稳定性

线程学到这里，你会自然遇到一个问题：

> 如果线程这么危险，为什么还要用？

答案是：

- **线程快**
- **进程安全**

### 1. 进程的工程意义

- 地址空间隔离
- 崩溃不扩散
- 权限控制

这在服务器、媒体、浏览器中极其常见。

### 2. C++ 学习重点（不是 STL，而是 OS）

- `fork / exec`
- 子进程回收
- 父子进程关系
- 僵尸进程 / 孤儿进程

你要意识到：

> 进程是“天然同步”的 —— 因为不共享内存。

------

## 五、第四阶段：进程间通信（IPC）——打破隔离但保持边界

当你有多个进程时，必然会遇到：

> 它们如何协作？

### 1. IPC 的分类理解（比 API 更重要）

| IPC 类型    | 特点           |
| ----------- | -------------- |
| 管道 / FIFO | 简单、单向     |
| 共享内存    | 快、但需要同步 |
| 消息队列    | 解耦、可靠     |
| Socket      | 通用、可跨机器 |

**关键认知：**

> IPC 本质是“受控共享”

你会发现：

- 共享内存 + mutex ≈ 多线程
- socket + 事件循环 ≈ 分布式线程

------

### 2. 为什么学 IPC 有助于多线程？

因为它会**反过来逼你思考边界**：

- 什么状态必须共享？
- 什么可以通过消息传递？
- 是否可以“避免共享”而不是“保护共享”？

这是高质量并发系统的核心思想。

------

## 六、最终学习路线总结（工程化）

### 阶段化路线

```
阶段 1：std::thread
  → 并发的存在

阶段 2：mutex / condition_variable
  → 并发的正确性

阶段 3：atomic / 线程池
  → 并发的性能

阶段 4：进程 / IPC
  → 系统级并发设计
```
