在 C++ 通用容器或算法中，对元素做“就地（in‑place）”操作时，如果元素类型 `T` 的操作（如拷贝、移动、交换）可能抛异常，就要认真考虑 **异常安全**。下面先回顾几个异常安全的级别，再针对 `reverse()` 中的“就地交换”提出替代策略，并举出更多例子。

------

## 一、异常安全级别回顾

1. **无抛保证（No‑throw guarantee）**
    函数在任何情况下都不会抛出异常。调用者无需做任何回滚或清理。
2. **强保证（Strong guarantee）**
    如果操作成功，则如期完成；如果操作过程中抛出异常，对象保持到操作之前的状态（要么全部完成，要么不做任何修改）。
3. **基本保证（Basic guarantee）**
    即使操作失败并抛出异常，对象保持处于一个**有效**但不一定和操作前相同的状态（不会泄漏资源、不会产生不一致的内部状态）。

------

## 二、针对 `reverse()` 的安全替代策略

### 1. 就地交换 —— 只能做到“基本保证”

```cpp
void Array<T>::reverse_inplace() {
    for (size_t i = 0; i + 1 < _size - i; ++i) {
        using std::swap;
        swap(ptr[i], ptr[_size - 1 - i]);  // 若 swap 抛，前面一部分已经改了
    }
    // 如果中途抛，数组部分被反转，部分未反转 → 有效但半完成（符合基本保证）
}
```

- **风险**：若第 k 次交换抛出，前 k–1 对元素已互换，而剩下的未交换。
- **级别**：**基本保证**——数组始终有效，但不再是“完整反转”或“未反转”状态。

### 2. 分配临时缓冲区 → 强保证

1. **分配** 新数组 `T* tmp = new T[_size]`
2. **拷贝** 元素到 `tmp` 时做反序
3. **交换指针** 或者用 `std::swap(ptr, tmp)`
4. **释放** 旧数组

```cpp
void Array<T>::reverse_strong() {
    // 1. 分配
    std::unique_ptr<T[]> tmp(new T[_size]);  // 若 bad_alloc 抛，就地数据未动 → 无抛保证

    // 2. 填充
    for (size_t i = 0; i < _size; ++i) {
        tmp[i] = ptr[_size - 1 - i];  // 若赋值抛，tmp 会被自动 delete[]；原 ptr 不动 → 强保证
    }

    // 3. 交换并释放
    std::swap(ptr, tmp);             // 不抛
    // tmp 离开作用域后自动 delete[] 原 ptr
}
```

- **优点**：只要新数组分配与拷贝都成功，才替换；中途抛出时旧数组保持不变 → **强保证**。

## 三、更多实际开发中的异常安全例子

### 1. `std::vector<T>::push_back`

- **行为**：如果当前容量足够，直接调用 `emplace_back` 构造元素（可能抛，旧元素不动 → 基本保证）。
- **重分配分支**：
  1. 分配更大内存
  2. 将旧元素移动或拷贝到新内存（若某次移动抛，则销毁已构造的、释放新内存；旧容器不动）
  3. 构造新元素
  4. 释放旧内存
- **级别**：**强保证**（要么全部完成新容量和新元素，要么旧 `vector` 保持原样）。

### 2. `std::string::append`

- 在需要扩容时，和 `vector` 类似，会先分配新缓冲区、拷贝原字符串、再拷贝追加内容，最后一次性替换缓冲区 → **强保证**。
- 如果追加的 `operator+=` 在中途抛，原字符串保持不变。

### 3. 容器的拷贝赋值（`operator=`）

- 传统做法（习题书常提）：

  ```cpp
  MyContainer& operator=(MyContainer const& other) {
      if (this != &other) {
          // 1. 拷贝构造临时副本
          MyContainer tmp(other);      // 若抛，*this 未动 → 强保证
          // 2. 交换
          swap(*this, tmp);            // 无抛
      }
      return *this;
  }
  ```

- **级别**：**强保证**，因为只有在拷贝成功后才交换。

### 4. 事务性文件写入

- **场景**：更新磁盘配置文件
- **做法**：
  1. 将新内容写入临时文件
  2. `fsync` 强制刷新
  3. 原子地 `rename(tmp, real)`
- **抛出时**：若写入／刷新抛，临时文件会被清理，但真实文件未动 → **强保证**。

### 5. 数据库操作事务

- **场景**：更新多条记录
- **做法**：开启事务 → 执行多条 SQL 语句 → 提交（commit）
- **回滚**：若中途抛（如主键冲突），自动回滚，库中数据保持一致 → **强保证**。

------

### 小结

- **就地修改**（如直接交换）一般只能给出“基本保证”——对象始终有效，但可能处于半完成状态。
- **使用临时副本或缓冲区** + **后置替换**，就能提供 **强保证**——要么操作全部成功，要么一切不变。
- 在设计自己的容器或算法时，根据需求和性能权衡这两种策略，明确在文档或接口中标注异常安全级别。
