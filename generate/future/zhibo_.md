## 一、当下音视频直播**正在使用的协议版图**

### 1️⃣ 从“位置”看协议（最重要的理解方式）

直播系统不是只用一种协议，而是**不同阶段用不同协议**：

| 位置          | 协议角色                 | 主流协议                               |
| ------------- | ------------------------ | -------------------------------------- |
| 主播 → 服务器 | **推流协议（Ingest）**   | RTMP（仍是主流）WebRTC（新兴，低延迟） |
| 服务器内部    | **内部流转**             | RTP / 私有二进制帧流                   |
| 服务器 → 观众 | **分发协议（Delivery）** | HLS、HTTP-FLV、RTMP、WebRTC            |

> **关键认知**：
> **RTMP ≠ HLS ≠ HTTP-FLV 的“替代关系”，而是“分工关系”**。

------

### 2️⃣ 现实世界里最常见的组合

**当前主流直播平台（国内外）大致是这样用的：**

- **主播端**：
  - OBS / 推流软件 → **RTMP 推流**
- **服务端**：
  - 接 RTMP → 内部转码/转封装 → 多协议分发
- **观众端**：
  - 浏览器 / App → **HLS 或 HTTP-FLV**
  - 超低延迟场景 → **WebRTC**

这是因为：

- RTMP **推流稳定、生态成熟**
- HLS **兼容性极强**
- HTTP-FLV **延迟低、实现简单**
- WebRTC **延迟最低，但复杂度最高**

## 二、直播中“真正传输的音视频格式”是什么？

这是一个**极其容易混淆、但必须一次讲清的问题**。

### 结论先行（非常重要）

> **直播中真正传输的不是“MP4 / FLV / TS”，而是：
> 👉 已编码的音视频数据（H.264 / H.265 / AAC 等）
> 👉 外面再套一层“协议/容器的壳”**

------

### 1️⃣ 编码格式（Codec）——“内容本体”

这是**音视频数据的本质形态**：

- **视频**
  - H.264（AVC）✅ 绝对主流
  - H.265（HEVC）⚠️ 带宽省，但兼容性/授权复杂
  - AV1 ⚠️ 前沿，但直播尚未全面普及
- **音频**
  - AAC ✅ 事实标准
  - Opus（WebRTC 常用）

> 这一步通常发生在：
> **摄像头 → 编码器（x264 / 硬编）**

------

### 2️⃣ 容器 / 封装格式（Container）——“打包方式”

容器只负责：

- 把 **编码帧** 组织起来
- 带上 **时间戳 / 类型 / 顺序**

| 场景     | 常见封装   |
| -------- | ---------- |
| RTMP     | FLV tag    |
| HTTP-FLV | FLV tag    |
| HLS      | TS 或 fMP4 |
| WebRTC   | RTP Packet |

**重点一句话**：
👉 **同一份 H.264/AAC 数据，可以同时被封装成 FLV、TS、fMP4**
👉 服务器做的事，往往是 **Remux（重封装）而不是转码**

#### **Step 1：采集与编码（主播侧）**

```
摄像头 / 麦克风
        ↓
视频帧 / 音频采样
        ↓
H.264 / AAC 编码
```

此时得到的是：

- 一帧一帧 **压缩后的数据**
- 带 PTS/DTS、关键帧（IDR）

------

#### **Step 2：推流（Ingest）**

```
编码帧
  ↓
FLV tag（封装）
  ↓
RTMP 推流
```

此阶段特点：

- **低延迟**
- **单连接**
- **主播 → 服务器**

------

#### **Step 3：服务端接入与统一流模型**

```
RTMP Server
    ↓
FLV tag / H264/AAC
    ↓
统一 MediaFrame / Packet
```

这是你作为**服务端开发者的核心位置**：

- 不解码
- 不重编码
- 只做：**解析、缓存、转发、重封装**

------

#### **Step 4：多协议分发（Fan-out）**

```
           ┌─→ HTTP-FLV（低延迟）
统一流 ────┼─→ HLS（高兼容）
           └─→ RTMP play / WebRTC
```

**同一条直播流，被“分叉”成多种协议**。

------

#### **Step 5：观众播放**

- 浏览器：
  - HLS → Safari / hls.js
  - HTTP-FLV → flv.js + MSE
- App：
  - 原生播放器 / SDK
- 超低延迟互动：
  - WebRTC

------

### 2️⃣ 用一句话总结整条链路

> **直播系统的核心不是“播放”，而是：
> 👉 如何稳定地接住一条编码流
> 👉 如何把它按时间、有序地分发给大量不同能力的客户端**

## 阶段 1：解构

### 1）一句话本质

**音视频直播服务开发的本质，是把“编码后的音视频帧”以正确的时间戳与关键帧边界组织起来，并通过不同协议（RTMP / HLS / HTTP-FLV）高效地分发给大量客户端。**

### 2）极简比喻

把直播想成“**现做现卖的流水席**”：

- 厨房不断出菜（编码器产出音视频帧）
- 你要把菜按顺序、按节奏端出去（时间戳/调度）
- 不同顾客用不同方式取餐：
  - RTMP：坐在吧台上即时端（低延迟、专用协议）
  - HTTP-FLV：窗口排队一边走一边吃（HTTP长连接、低延迟）
  - HLS：打包成便当、定时上架（分片+清单、兼容性强但延迟更高）

------

### A1. 要素：最关键的“零件”

围绕你列的 5 个目标，可以把直播系统拆成 8 个核心零件（从原理出发，不依赖某个具体库）：

1. **媒体表示（Media Model）**

- 统一的数据抽象：`VideoFrame/AudioFrame`（payload + pts/dts + keyframe标记 + track信息）
- 关键：**时间戳语义**（PTS/DTS）、**关键帧边界**（IDR）、**GOP结构**

1. **接入（Ingest）**

- 你要“接住”推流：最常见是 **RTMP publish**
- 关键：RTMP 握手、Chunk、AMF 命令、音视频消息

1. **解复用/复用（Demux/Mux，重封装 Remux）**

- 直播服务端一般不解码不编码，更多是 **容器/协议层重封装**：
  - RTMP 输入（FLV tag）→ 输出 HTTP-FLV（还是 FLV tag）
  - RTMP 输入（H264/AAC）→ 输出 HLS（TS 或 fMP4 分片）
- 关键：**不改码流，只改“包装”**

1. **分发枢纽（Stream Hub / Fanout）**

- 一路流对应一个“广播站”：一个生产者（推流）→ 多个消费者（播放）
- 关键：**多订阅者扇出、背压策略、内存复用、慢客户端处理**

1. **协议输出（Egress）**

- HLS：生成 `m3u8` + `ts/fmp4` 分片并提供 HTTP 下载
- HTTP-FLV：HTTP长连接持续写出 FLV header + tags
- RTMP play：RTMP 会话内发送音视频消息

1. **时间与缓冲（Pacing & Buffering）**

- 你必须决定“什么时候发送下一帧/下一片”
- 关键：**以DTS/PTS为准的节奏控制、抖动缓冲、关键帧对齐**

1. **服务端网络模型（I/O & Concurrency）**

- 高性能 HTTP-FLV 的核心通常不在“协议语法”，而在：
  - epoll/异步 I/O
  - 零拷贝/少拷贝
  - 内存池
  - per-stream 单线程或分片锁策略

1. **可观测与运维（Metrics/Logs/Health）**

- 直播最怕“看起来在跑但其实掉帧/延迟累积”
- 关键：延迟、丢帧、关键帧间隔、在线人数、发送队列积压

------

### A2. 结构：零件如何连接、遵循什么规则

可以用一条“主干链路 + 三种出口”理解：

**主干链路规则（最重要的三条）**

1. **时间戳规则**：任何输出协议都要维持“单调递增的时间线”（至少对DTS单调），否则播放器会卡顿/回退。
2. **关键帧规则**：新加入的播放端必须从关键帧开始拿数据（否则无法解码）。
3. **背压规则**：慢客户端不能拖垮系统——必须丢弃/断开/降级（例如只保留最近关键帧后的窗口）。

**结构关系（从数据流视角）**

- Ingest 把数据变成统一 `MediaFrame`（或统一的“Tag/Packet”）
- Stream Hub 保存“可供新订阅者快速起播”的关键数据（如：最近IDR + 一小段后续帧）
- Egress 按各自协议包装并发送

------

### A3. 系统：组合后涌现的核心功能/最终目的

当以上零件按规则组合起来，系统涌现的核心能力是：

- **持续接入一条直播流**
- **支持多协议分发（HLS、HTTP-FLV、RTMP）**
- **让新观众能“快速起播”且播放稳定**
- **在高并发下保持可控的CPU/内存与延迟曲线**

## 场景 1：电商直播间“秒级互动”看货下单（HTTP-FLV + 高性能）

### B1. 设定目标

在“电商带货直播间”，用户要求“弹幕互动、秒级反馈”。目标：

- 实现最简单的 **HTTP-FLV 服务器**
- 浏览器端用 **MSE + flv.js** 播放
- 进一步演进为 **高性能 HTTP-FLV（多用户、低CPU、低内存抖动）**

### B2. 应用规则（对应阶段1）

- 时间戳规则：FLV tag timestamp 必须单调递增（至少不乱序）
- 关键帧规则：新观众必须先拿到“最近关键帧（IDR）”再继续
- 背压规则：慢连接不允许无限堆积，采用“滑动窗口/丢弃策略”

### B3. 创造方案：最小可行模型（MVP）

**MVP 服务器可以拆成三块：**

1. **Ingest（先不自己写RTMP）**：用 FFmpeg 把本地文件或摄像头推成 FLV tag 流，喂给你的 Hub（后续再替换成自研 RTMP）。
2. **Stream Hub（每路流一个Hub）**：
   - 缓存：`flv_header` + `last_idr_tag` + `recent_tags_ringbuffer`
   - 订阅：HTTP连接加入后，先发 header + last_idr + ringbuffer，再进入实时转发
3. **HTTP-FLV 输出**：
   - `Content-Type: video/x-flv`
   - 首包写 FLV header
   - 后续不断写 tags（Chunked Transfer 或直接 keep-alive 写）

**高性能演进点（只讲原理，不陷入细节）：**

- 单份 tag 内存，多订阅者共享引用（引用计数/共享块），避免 N 份拷贝
- per-stream 单线程事件循环（或按流哈希分片到多个 loop），避免全局大锁
- 慢客户端：当其发送队列超过阈值，直接丢弃旧tag或断开

### B4. 具象图示（ASCII）

```
             (推流/输入：先用 FFmpeg 或后续 RTMP publish)
                         |
                         v
                  +--------------+
                  |   Ingest     |
                  |  (FLV tags)  |
                  +--------------+
                         |
                         v
           +--------------------------------+
           |        Stream Hub (per stream) |
           |  - cache: FLV header           |
           |  - cache: last IDR tag         |
           |  - ringbuffer: recent tags     |
           |  - subscribers: conn list      |
           +--------------------------------+
                 |                |                 |
                 v                v                 v
        +---------------+  +---------------+  +---------------+
        | HTTP-FLV conn |  | HTTP-FLV conn |  | HTTP-FLV conn |
        | 先发header     |  | 先发header     |  | 先发header     |
        | 再发last IDR   |  | 再发last IDR   |  | 再发last IDR   |
        | 再进入实时转发 |  | 再进入实时转发 |  | 再进入实时转发 |
        +---------------+  +---------------+  +---------------+

背压策略（每个conn）:
send_queue 超阈值 -> 丢弃旧tag/断开 -> Hub不被拖慢
```

## 场景 2：高校运动会“全校都能看”（HLS 最小服务器 + 兼容性优先）

### B1. 设定目标

“运动会直播”要求“任何手机都能看”，延迟可以稍高但必须稳定。目标：

- 实现最简单的 **HLS 服务器**（Live）
- 浏览器播放 HLS（Safari 原生 / 其它浏览器用 hls.js）
- 重点：分片与播放列表的正确性

### B2. 应用规则

- 关键帧规则：**分片边界最好对齐关键帧**，否则某些播放器起播慢或花屏
- 时间戳规则：TS/fMP4 内部时间线必须连续/单调
- 滑动窗口规则：Live m3u8 是“滚动窗口”，旧分片按策略淘汰

### B3. 创造方案：最小可行模型（MVP）

**最小 HLS Live 服务器（原则性模型）**

1. 输入：仍先用 Ingest 得到 H264/AAC 的访问单元（或直接使用 FFmpeg 产出原始 AnnexB/AAC）。
2. Segmenter（分片器）：
   - 按固定时长（如 2s/4s）切片
   - 切片点尽量落在视频关键帧（IDR）上
3. Playlist（m3u8 生成器）：
   - 维护 `MEDIA-SEQUENCE`
   - 维护最近 N 个分片列表
4. HTTP Serve：提供 `index.m3u8` + `segXXXX.ts`（或 `.m4s`）

**浏览器播放要点（原理层面）**

- Safari：通常可直接播放 m3u8
- Chrome/Edge：需要 MSE + hls.js（本质是“JS拉取分片并喂给MSE”）
- CORS/Cache-Control：影响加载稳定性与刷新策略（尤其 live）

     -  `tcp_server/acceptor`：监听端口，接受新连接，为每个客户端创建 Connection
     -  `Buffer`：可增长的读写缓冲区，支持 append、retrieve、peek 等基础操作
  2. RTSP 协议层（rtsp 模块）
     -  `RtspParser`：从 Buffer 中解析出 `RtspRequest`
     -  `RtspDispatcher`：根据 Method 分发给不同 handler
     -  `IRtspHandler` + 各种 Handler：
       - OPTIONS handler
       - DESCRIBE handler（构造 SDP）
       - SETUP handler（分配 session、端口、transport）
       - PLAY handler（触发媒体发送）
     -  `RtspResponseBuilder`：统一构造响应报文
  3. 媒体层（media 模块，哪怕现在只是 MVP）
     -  `MediaSourceManager`：管理可用的媒体源（本地 mp4 文件列表 / URL 映射）
     -  `FileMediaSource`：基于 FFmpeg 打开 mp4 文件，获取媒体信息（分辨率、编码、时长等）
     -  媒体信息结构：`MediaInfo / TrackInfo`（视频轨、音频轨基本信息）
     -  RTP builder MVP：能够把单个 H.264 NALU / AAC frame 封装成一帧 RTP 包（先不追求复杂分片）
  4. RTP / 播放链路
     -  与客户端：用 `ffplay rtsp://...` 能稳定播放
     -  支持多个客户端同时连入、播放同一个本地文件
     -  Session 生命周期管理：SETUP 分配，TEARDOWN 或超时清理
  5. 基础工程能力
     -  清晰的目录结构：`net / rtsp / media / core` 等模块边界
     -  CMake 构建拆分（库 + 最终可执行程序）
     -  简单日志（至少 INFO/ERROR 输出）
     -  `.clang-format` 基本统一风格

  你可以在总结里把每一项标注成：

  - 已完成：说明具体是怎么实现的（关键类 + 关键流程）
  - 未完成但设计清晰：简单描述设计思路
  - 暂不考虑：说明为什么（比如放到 v2/v3）

  ### 3. v1 的架构图 / 文字描述（建议在总结里写）

  可以用一句话 + 一个简单数据流来描述整体：

  > epoll 事件驱动 → Connection 收发字节流 → RTSP 模块解析/生成信令 → Media 模块提供帧数据 → RTP builder 封装并通过 UDP/TCP 发送给客户端。

  你可以在文档里画 3 条主链路：

  1. 控制链路（RTSP）
  2. 媒体链路（MediaSource → RTP → Socket）
  3. 会话管理（Session / Connection 生命周期）

  ## 二、v2：性能版本（单机能力最大化）的项目级规划

  v2 的关键词是：**可量化的性能目标 + 有针对性的优化方向**。

  ### 1. v2 版本目标（Project Level）

  > 在单机环境下，把 RTSP 服务器的“吞吐能力”和“资源利用率”做到可观测、可优化：
  > 在固定硬件上，支持尽可能多的并发播放会话，同时保持 CPU/内存占用在合理范围内，无明显卡顿。

  建议你先给 v2 定几个“可度量”的指标（可以在规划文档中预留）：

  - 单机最大并发会话数：如 N 路 720p@30fps 播放
  - CPU 占用：在 N 路并发下 < 80%
  - 内存占用：控制在 M GB 内
  - 平均帧发送间隔误差：如 ≤ 5ms（和理想 33ms 或 40ms 相比）

  ### 2. v2 的核心方向

  可以分 4 个维度来规划任务：

  #### 2.1 事件模型 & 线程模型

  目标：**避免所有阻塞操作卡住主 reactor**。

  规划任务：

  1. 明确线程模型：
     - 方案 A：单 Reactor + 多 worker 线程（媒体发送、FFmpeg 相关工作）
     - 方案 B：多 Reactor（主线程 accept，新连接分发到不同 IO 线程的 epoll）
  2. 隔离阻塞操作：
     - 文件读取/解复用（FFmpeg）放到专门线程 / 线程池中
     - 网络发送（sendto/sendmsg）尽量在 IO 线程内，但要避免长时间卡住
  3. Connection 设计扩展：
     - 支持注册回调：read 事件 → RTSP 流程；定时器事件 → 媒体帧发送
     - 支持“轻量级任务队列”（比如将需要写的 RTP 包压入队列，在合适时机 flush）

  #### 2.2 内存与 Buffer 优化

  目标：**降低内存分配和拷贝次数**。

  规划任务：

  1. Buffer 设计升级：
     - 明确“读指针 / 写指针”
     - 支持 `readv/recv` 直接写入尾部，减少临时 buffer
  2. RTP 包构建策略：
     - 尽量避免“拷来拷去”：构建 header + payload 时，尽可能一次性写入 Buffer
     - 引入“对象池”：RTP 包结构、临时缓冲区可重用
  3. 分析热点分配：
     - 通过 `valgrind/massif` 或 `perf` 看哪些地方频繁 new/delete
     - 把这些地方抽象成池化资源

  #### 2.3 系统调用与 I/O 调优

  目标：**减少系统调用次数、提升数据通路效率**。

  规划任务：

  1. 合并写：
     - 研究 `sendmsg` / `writev` 或 `sendmmsg`，一次发送多个 RTP 包
  2. Socket 参数：
     - 调整 `SO_SNDBUF`、`IP_TTL` 等参数
     - 对 TCP 连接（如果你支持 interleaved RTP over RTSP）配置 `TCP_NODELAY`（视具体需求）
  3. epoll 使用：
     - 明确使用 LT / ET 策略
     - 确保每次事件处理都“读到 EAGAIN”，避免重复触发

  #### 2.4 性能测试与 Profiling 体系

  目标：**不是“感觉变快了”，而是用数据证明**。

  规划任务：

  1. 基准测试方案：
     - 脚本化：自动启动 N 个 `ffplay` 或自写 RTSP client
     - 记录 CPU、内存、平均帧间隔、丢帧情况
  2. Profiling 工具：
     - `perf` / `gprof` / `valgrind` 测热点函数
     - 用 flame graph 分析 CPU 消耗在哪些函数
  3. 性能报告模板：
     - 记录：测试环境（CPU/内存/OS）、版本号、配置、结果
     - 每次优化前后作对比

  ### 3. v2 的里程碑建议

  可以在 TODO 中设三步：

  1. M1：搭好性能测试脚本，跑出“当前 v1 上的 baseline 数据”
  2. M2：完成线程模型 + Buffer 优化，baseline 提升一轮
  3. M3：完成系统调用与参数调优，形成第一版性能报告

  ## 三、v3：真实网络环境 + 复杂场景的项目级规划

  v3 关键词：**复杂网络 + 部署运维 + 可靠性**。

  ### 1. v3 版本目标

  > 将服务器部署到真实物理/云服务器上，在复杂网络环境（跨公网、NAT、丢包、抖动、弱网）下，保持播放体验可接受，并为后续运维与扩展打下基础。

  ### 2. v3 的关注维度

  可以拆成 4 大块：

  #### 2.1 网络健壮性与协议完善

  1. 超时与心跳：
     - RTSP 会话超时（Session Timeout）
     - 空闲连接回收
  2. RTCP 支持（如果你打算深入）：
     - 发送 SR（Sender Report），正确绑定 NTP 时间戳和 RTP 时间戳
     - 解析客户端 RR（Receiver Report），感知丢包、抖动
  3. 重传与错误处理策略：
     - 对 UDP 丢包的态度：是完全忽略，还是做简单的“关键帧重发”（可作为高级目标）
  4. 多种传输模式：
     - UDP RTP
     - TCP interleaved（弱网/NAT 环境更常见）

  #### 2.2 部署、配置与可观测性

  1. 配置系统：
     - 支持配置端口、本地媒体路径、最大连接数、日志等级等
     - 配置文件（YAML/JSON） + 命令行参数
  2. 日志与指标：
     - 结构化日志（方便后期用工具收集）
     - 基础指标：QPS、当前会话数、平均发送 bit rate、错误数
  3. 健康检查与监控：
     - 提供简单的 HTTP 状态页（可选）
     - 或者至少有一个内部命令可以 dump 当前 session 状态

  #### 2.3 真实网络环境测试

  1. 场景设计：
     - 同城机房内测试
     - 跨运营商测试（移动/联通/电信）
     - 弱网仿真（限速、延迟、丢包，用 tc/netem 来模拟）
  2. 客户端多样性：
     - ffplay
     - VLC
     - 可能的硬件设备（如果后续有需求）
  3. 兼容性问题记录：
     - 不同客户端对 SDP、Transport 头的兼容差异
     - 不同网络环境下的表现

  #### 2.4 安全与访问控制（可以作为 v3 的后半程）

  1. 认证：
     - 最简单的 Basic Auth（用户名密码）
     - 如果有兴趣可以加 Digest Auth
  2. 加密：
     - RTSPS（RTSP over TLS）
     - 或者在内网先不做，加到 v3.5 / v4 也可以

  ### 3. v3 的里程碑建议

  1. M1：把当前服务部署到真实服务器，支持基本公网访问，记录第一轮问题
  2. M2：补齐超时、会话管理、简单 RTCP、日志指标，完成“生产可用的基础版”
  3. M3：针对几类典型网络场景（弱网、跨运营商）做专项测试，形成问题清单 + 解决方案
