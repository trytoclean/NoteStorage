## 1. 硬件架构的责任

CPU 硬件必须提供最基础的「模式切换能力」，主要包括：

- **特权模式**：用户态 (U-mode)、内核态 (S-mode)、机器态 (M-mode)。
- **特定寄存器**：
  - `sepc` → 保存异常发生的用户态 PC。
  - `stvec` → 保存 trap 入口地址。
  - `scause` → 保存异常/中断的原因。
  - `sstatus` → 保存 CPU 状态位（包含当前模式）。
- **特定指令**：
  - `ecall` 触发陷入。
  - `sret` 从 S-mode 返回 U-mode。
- **硬件保证**：执行 `ecall` 时自动保存 PC、切换模式、跳转到 `stvec`。

👉 硬件只提供「最小机制」，并不关心你内核怎么组织代码。

------

## 2. 操作系统的责任

操作系统利用这些机制来实现「系统调用」和「上下文切换」。

- 设置好 `stvec` → 告诉 CPU 出错时跳到哪里。
- 写好 trap 入口的汇编代码 → 保存寄存器、切换栈。
- 写好 trap 处理函数（C 代码）→ 分发系统调用。
- 写好返回路径代码 → 恢复寄存器，执行 `sret` 回到用户态。

👉 操作系统是「用硬件的接口拼装出一个完整的用户态 ↔ 内核态切换流程」。

------

## 3. trampoline 的定位

- **不是硬件强制的**。
- **是 xv6 的实现细节**：
  - 单独放一页 trampoline 代码。
  - 既能被用户页表映射，也能被内核页表映射。
  - 统一处理「陷入」和「返回」的汇编逻辑（`uservec`、`userret`）。

👉 在 Linux RISC-V 或其他系统里，也有类似功能的代码，但布局和命名不一样。

------

## 4. 一句话总结

你说的这句话非常准确，可以概括为：

**「硬件架构提供寄存器和指令，保证模式切换最小能力；操作系统利用它们设计具体机制。trampoline 是 xv6 的一种实现方式，不是硬件特性。」**
