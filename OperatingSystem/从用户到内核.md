## 1. 抽象、通用的「用户态 → 内核态切换」流程

无论是 **RISC-V、x86、ARM**，都有这么一条共性主线：

1. **用户态程序请求服务**
   - 比如调用 `write(fd, buf, len)`。
   - 用户代码本身不能直接访问硬件或内核数据结构。
2. **触发陷入（Trap）**
   - 通过专门的指令进入内核态：
     - RISC-V → `ecall`
     - x86 → `int 0x80`、`syscall/sysenter`
     - ARM → `svc`
   - CPU 发现这是特权操作 → **切换到内核态**。
3. **硬件动作（保护与准备）**
   - 保存用户态的 PC、部分寄存器。
   - 设置 CPU 模式位（U → S / R3 → R0）。
   - 跳转到异常处理入口（由内核提前设置）。
4. **内核的 Trap 处理程序**
   - 保存用户寄存器上下文（软件部分）。
   - 解析系统调用号（来自寄存器/栈）。
   - 分发到具体的内核函数（如 `sys_write`）。
5. **执行内核代码**
   - 内核完成系统调用请求（比如写数据到文件描述符）。
6. **返回用户态**
   - 内核恢复寄存器上下文。
   - 将结果放入寄存器（如 `a0`/`rax`）。
   - 执行特权指令返回（RISC-V → `sret`，x86 → `iretq`，ARM → `eret`）。
   - CPU 切回用户态，从原来保存的 PC 继续执行。

------

## 2. RISC-V 的具体实现

你之前写的就是把上面这条「抽象流程」落在了 RISC-V 上：

- 特殊寄存器：`stvec`、`sepc`、`sstatus`。
- 汇编入口：`uservec`。
- C 语言分发：`usertrap()`。
- 返回：`userret` → `sret`。

也就是说：

- **抽象层面**：用户态不能直接访问内核资源 → 必须通过 trap 进入内核 → 执行完再返回。
- **具体层面**：不同架构只是换了触发指令和具体寄存器名而已。

------

## 3. 你的学习路线建议

如果你目标是「理解用户态到内核态切换」，建议这样分层：

- **第一层：抽象模型**（任何 CPU 都适用）
   用户态 → 触发 trap → 内核处理 → 返回用户态。
- **第二层：架构细节**（RISC-V vs x86 vs ARM）
   哪条指令触发？哪些寄存器保存上下文？返回用哪条指令？
- **第三层：操作系统实现**
   xv6 (RISC-V)、Linux (x86/ARM) 怎么把这些硬件能力拼接成「系统调用接口」。
