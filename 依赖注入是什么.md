先把几个词拆开。很多人第一次看到 **依赖注入（Dependency Injection, DI）** 会觉得像魔法，其实只是“谁创建对象”的问题。

核心概念只有三个：

**1 依赖（dependency）**

如果一个类在工作时必须使用另一个类，它就“依赖”那个类。

例子：

```cpp
class Logger {
public:
    void log(const std::string& s);
};

class Service {
public:
    void work() {
        logger.log("run");
    }

private:
    Logger logger;
};
```

这里：

```
Service 依赖 Logger
```

因为 `Service` 需要 `Logger` 才能工作。

------

**2 隐式依赖（implicit dependency）**

看一个常见写法：

```cpp
class Logger {
public:
    static Logger& instance() {
        static Logger logger;
        return logger;
    }
};

class Service {
public:
    void work() {
        Logger::instance().log("run");
    }
};
```

依赖关系：

```
Service → Logger
```

问题是：

这个依赖 **没有出现在接口里**。

`Service` 的定义：

```
Service()
```

看起来不需要任何东西，但内部偷偷用了 `Logger`。

这叫：

```
隐藏依赖
implicit dependency
```

在小程序里没问题，在大型系统里就会变成一团线。

------

**3 为什么测试困难**

想做单元测试：

目标：

```
测试 Service
```

但 `Service` 内部写死：

```
Logger::instance()
```

你无法替换 Logger。

测试时你可能想要：

```
FakeLogger
TestLogger
MockLogger
```

比如：

```cpp
class FakeLogger {
public:
    void log(const std::string& s) {
        messages.push_back(s);
    }

    std::vector<std::string> messages;
};
```

如果代码写死 `Logger::instance()`，替换不了。

测试就被锁死。

------

# 依赖注入是什么

DI 的意思：

```
对象不要自己创建依赖
依赖由外部提供
```

改写代码：

```cpp
class Service {
public:

    Service(Logger& logger)
        : logger_(logger) {}

    void work() {
        logger_.log("run");
    }

private:
    Logger& logger_;
};
```

现在依赖关系：

```
Service(Logger&)
```

写在构造函数里。

使用：

```cpp
Logger logger;
Service service(logger);
```

这就叫：

```
dependency injection
```

依赖被 **注入** 进去。

------

# DI 的本质

一句话：

```
谁负责创建对象？
```

两种模式：

### 方式1（单例）

类自己找依赖

```
Service -> Logger::instance()
```

关系：

```
Service
  |
  v
Logger
```

------

### 方式2（DI）

外部提供依赖

```
main
 |
 v
Service(Logger&)
```

结构：

```
main
  |
  v
Service ----> Logger
```

控制权在外部。

------

# 在服务器工程里的差别

假设你在写流媒体服务器：

模块：

```
RTMPServer
SessionManager
Logger
Config
Metrics
```

如果全部单例：

```
Session -> Logger::instance()
RTMP -> Logger::instance()
HTTP -> Logger::instance()
```

每个模块都直接抓 Logger。

系统耦合变成蜘蛛网。

而 DI 结构：

```
main
 ├── Logger
 ├── Config
 └── Server
        ├── RTMP
        └── HTTP
```

依赖关系清晰。

------

# 为什么大型系统更喜欢 DI

原因只有三个：

**1 可测试**

可以替换依赖。

```
Logger → FakeLogger
DB → MockDB
```

------

**2 可扩展**

日志可以换实现：

```
FileLogger
ConsoleLogger
RemoteLogger
```

------

**3 依赖关系清晰**

只看构造函数就知道需要什么。

------

# 但现实工程并不是“只用 DI”

常见组合：

```
Logger        -> Singleton
ThreadPool    -> Singleton
Config        -> Singleton
Service       -> DI
Session       -> DI
```

因为：

有些东西 **逻辑上就是全局唯一资源**。

例如：

```
日志
线程池
metrics
```

单例反而更自然。

------

最后给一个工程经验层面的结论：

**Singleton 解决“唯一资源”。**
**DI 解决“模块解耦”。**

它们不是互斥关系，而是不同层级的工具。

如果继续深入，会看到一个更有趣的结构：
大型服务器（包括很多流媒体系统）其实是 **DI + Singleton + Service Locator 的混合体**。这背后是一套“依赖图”的工程哲学。
